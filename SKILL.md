# Content Adapter - 多平台文案适配器

## Overview

将一篇原始内容智能改写为多个社交平台风格的文案。AI 根据各平台的用户画像和表达习惯，真正改写文风和结构，而非简单模板替换。

**核心原则：去模板化，像真人在说话。** 改写后的文案应该有真实感、有态度、有画面感，而不是填表式的公文。

**风格指南数据来源：** MediaCrawler（GitHub 30K+ star 多平台爬虫）采集的真实爆款内容 + 各平台运营数据分析报告 + chinese-copywriting-guidelines 中文排版规范。

## When to Trigger

当用户提到以下意图时激活此 Skill：
- 将内容改写/适配/转换为某个平台风格
- 多平台分发/发布/投稿
- 生成小红书/公众号/知乎/B站/微博/抖音文案
- 为内容生成不同风格的标题
- 内容运营、社媒文案相关需求

## Workflow

### Step 1: 内容分析

收到用户内容后，先调用脚本分析内容特征：

```bash
node "scripts/adapter.js" analyze --input "<原始文本>"
```

返回内容类型、关键词、核心信息点、情感基调、推荐平台等分析结果。

### Step 2: 读取详细风格指南

**关键步骤：** 对每个目标平台，必须读取 `references/` 目录下对应的详细风格指南文件，获取完整的爆款标题公式、正文结构模板、真实案例和排版细节。

| 平台 | 代码 | 风格指南文件 |
|------|------|-------------|
| 小红书 | red | `references/red-style-guide.md` |
| 微信公众号 | wx | `references/wx-style-guide.md` |
| 知乎 | zh | `references/zh-style-guide.md` |
| B站 | bili | `references/bili-style-guide.md` |
| 微博 | wb | `references/wb-style-guide.md` |
| 抖音 | dy | `references/dy-style-guide.md` |

**必须读取对应文件后再改写，不能仅凭下方概要信息改写。**

### Step 3: 确认目标平台和需求

根据用户指定的平台（未指定则默认全部6个平台），确认：
- 目标平台列表
- 是否需要标题生成
- 是否有字数限制
- 是否有特定语气偏好

### Step 4: AI 智能改写

读取详细风格指南后，对每个目标平台进行 AI 改写。

**改写原则：**
1. 保持原始内容的核心信息和事实不变
2. 根据平台用户画像调整语气、用词、结构
3. 标题必须贴合平台调性，不能通用——每个平台的标题风格完全不同
4. emoji、标签、排版符合平台惯例
5. 不夸大、不误导、不编造原文没有的信息
6. 参考风格指南中的真实爆款案例，模仿其结构和语气
7. 使用风格指南中数据验证的标题公式生成标题

**🚫 反模板规则（必须遵守）：**
- 不要用"绝了！""姐妹们！"这种烂大街的开头，除非原文本身就是这种风格
- 不要每个卖点都机械地对应一个emoji（💡卖点1 📌卖点2 ✨卖点3），要有节奏感
- 不要用"## 一、## 二、## 三"这种机械的小标题结构，用更有故事感的标题
- 不要用"近年来研究表明""笔者认为"这种填充语，直接说结论
- 不要所有平台都用同样的标题模板，每个平台的标题必须完全不同
- 不要无脑吹产品，要有态度——可以承认局限性，可以说"还没到完美"
- 用具体场景代替抽象描述（"够你看两部半电影" 比 "节省3小时" 有画面感）
- 语气像真人在说话，不是AI生成的公关稿或说明书

**✅ 高质量改写的特征：**
- 有真实感：像在分享经验，不是在念稿
- 有态度：不是无脑吹，可以承认不足
- 有画面感：用具体场景让读者有代入感
- 有节奏：长短句搭配，不是机械的列表

### Step 5: 改写质量自检

改写完成后，对每个平台的输出进行质量自检：

**自检清单：**
- [ ] 核心信息是否完整保留（对比原文的关键信息点）
- [ ] 语气是否符合平台特征（小红书种草感/知乎专业感/B站轻松感等）
- [ ] 标题是否使用了对应平台的标题公式（而非通用标题）
- [ ] 字数是否在推荐范围内
- [ ] emoji 密度是否符合平台习惯
- [ ] 排版是否符合平台惯例（标签格式、段落长度等）
- [ ] 是否有编造原文不存在的信息
- [ ] **是否有模板感？（机械emoji列表/##一二三结构/填充语/烂大街开头）——有则重改**
- [ ] **是否有真实感？（像真人在说话，不是AI公文）——没有则重改**

如果自检不通过，立即修正后再输出。

### Step 6: 格式化输出

改写和自检完成后，调用脚本生成结构化输出：

```bash
node "scripts/adapter.js" format --platform <平台代码> --title "<标题>" --content "<改写后文案>" --output <输出路径>
```

### Step 7: Web 预览（可选）

生成漂亮的6平台对比 HTML 页面，方便截图展示效果：

```bash
node "scripts/preview.js" --data <JSON文件> [输出路径]
```

如果用户需要多平台批量导出：

```bash
node "scripts/adapter.js" export --results '<JSON结果>' --output-dir <目录>
```

### Step 8: 呈现结果

按下方 Output Format 呈现每个平台的改写结果。

## Platform Style Guide (概要)

> ⚠️ 以下仅为各平台风格概要。改写时必须读取 `references/` 下对应的详细风格指南文件，获取完整的标题公式、结构模板、真实案例和排版细节。

### 小红书 (RED)

**用户画像：** 18-35岁女性为主，追求生活品质，喜欢被"种草"
**语气：** 种草体、分享体、闺蜜感——但要有真实感，不是广告
**字数：** 150-300字
**emoji密度：** 有，但有节奏，不要每行一个
**结构：** 吸引开头 + 分点说明 + 行动号召 + 标签
**标题公式：** 痛点共鸣型 > 结果前置+场景限定+低门槛 / 人群+痛点+利益点 / 反差+核心信息
**详细指南：** `references/red-style-guide.md`

### 微信公众号 (WX)

**用户画像：** 25-45岁，追求深度内容，习惯长文阅读
**语气：** 专业温暖、有深度、共情力——像朋友在分享经验，不是说教
**字数：** 800-2000字
**emoji密度：** 低或无
**结构：** 引语/故事 + 分段正文(小标题) + 结语
**标题公式：** 故事引入型 > 悬念型 / 反常识型 / 数字清单型
**详细指南：** `references/wx-style-guide.md`

### 知乎 (ZH)

**用户画像：** 20-40岁，高学历，追求专业深度和逻辑分析
**语气：** 专业分析、客观理性、数据支撑——可以有态度，承认局限性
**字数：** 1000-3000字
**emoji密度：** 无
**结构：** 问题引入 + 结论先行 + 详细分析 + 建议
**标题公式：** 身份+实测型 > 评价型 / 质疑型 / 对比型 / 深度型
**详细指南：** `references/zh-style-guide.md`

### B站 (BILI)

**用户画像：** 16-30岁，Z世代为主，喜欢轻松有趣的内容
**语气：** 轻松幽默、口语化、互动感强——像UP主在测评，不是广告
**字数：** 200-500字
**emoji密度：** 中等
**结构：** 梗开头 + 正文(括号吐槽) + 互动结尾
**标题公式：** 实测+悬念型 > 情绪爆发型 / 挑战悬念型 / 安利攻略型
**详细指南：** `references/bili-style-guide.md`

### 微博 (WB)

**用户画像：** 18-40岁，信息获取快，关注热点话题
**语气：** 直接有力、话题性、信息密度高——事实说话
**字数：** 50-140字
**emoji密度：** 低
**结构：** #话题标签# + 核心观点 + 补充说明
**标题公式：** 话题+数据型 > 话题+观点型 / 话题+悬念型 / 直击型
**详细指南：** `references/wb-style-guide.md`

### 抖音 (DY)

**用户画像：** 18-35岁，短视频消费为主，注意力短
**语气：** 口语化、节奏快、有画面感——像朋友在聊天
**字数：** 50-150字（视频脚本/文案）
**emoji密度：** 低
**结构：** 3秒钩子 + 核心信息 + 行动号召
**标题公式：** 反差+数字型 > 悬念钩子型 / 反转型 / 共鸣型 / 数字型
**详细指南：** `references/dy-style-guide.md`

## Advanced Features

### 内容类型自动识别

脚本可自动识别内容类型，帮助 AI 更精准地改写：

```bash
node "scripts/adapter.js" analyze --input "<文本>"
```

返回：内容类型（产品/活动/知识/观点/故事）、情感基调、核心信息点列表、推荐平台

### 改写质量自检

每次改写后自动执行质量自检（见 Step 5），确保输出质量。自检不通过则自动修正。

### 批量导出

将所有平台结果导出为 Markdown 文件：

```bash
node "scripts/adapter.js" export --results '<JSON结果>' --output-dir <目录>
```

### 自定义风格

用户可以指定额外要求，如：
- "语气更活泼一点"
- "面向大学生群体"
- "突出性价比"
- "不要用emoji"

AI 应将这些要求融入改写过程，并在质量自检中额外验证。

## Notes

- 改写不是翻译，是重新创作——保持信息不变，但表达方式要完全贴合平台
- 标题是各平台的第一印象，必须精心设计，不能通用
- 每个平台的用户期待不同：小红书要"被种草"，知乎要"被说服"，B站要"被逗乐"，抖音要"被吸引"
- 如果原文信息量不足以支撑某个平台的目标字数，应适当扩展而非注水
- 所有改写都遵循原意，不夸大、不误导、不编造
- 必须读取 references/ 下的详细风格指南后再改写，不能仅凭概要信息
- **最重要的一条：写出来的东西要让读者觉得是真人写的，不是AI生成的。有态度、有真实感、有画面感，比完美更重要。**
